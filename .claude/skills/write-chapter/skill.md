# 챕터 집필 스킬

챕터 설계서(spec)와 소스 포스트를 기반으로 하나의 챕터를 deep rewrite하여 작성하는 스킬.

## 입력 파라미터

| 파라미터 | 필수 | 기본값 | 설명 |
|----------|------|--------|------|
| chapter_number | O | - | 챕터 번호 |
| spec_path | O | `_workspace/book/blueprint/chapter-specs/ch-{NN}-spec.md` | 챕터 설계서 경로 |
| blueprint_path | O | `_workspace/book/blueprint/book-blueprint.md` | 책 전체 구조 경로 |
| revision_feedback | X | - | 편집자 반려 사유 및 수정 지시 |

## 실행 절차

### 1단계: 맥락 파악

1. `book-blueprint.md`를 읽어 전체 서사 아크와 톤 가이드를 파악한다
2. 이 챕터가 전체에서 어떤 위치에 있는지(제N막, 앞뒤 챕터)를 인지한다

### 2단계: Spec 분석

`ch-{NN}-spec.md`를 읽고 다음을 확인한다:
- 핵심 논지
- 소스 포스트와 사용할 부분
- 구성 (도입/전개/마무리)
- 전환 지침 (앞뒤 챕터와의 연결)
- 주의사항

### 3단계: 소스 포스트 읽기

spec에 명시된 소스 포스트들을 읽고, 각 포스트에서:
- 핵심 아이디어를 추출한다 (문장이 아닌 개념 수준)
- 재사용할 구체적 사례/경험을 메모한다
- 메타포와 비유를 식별한다

### 4단계: 챕터 본문 작성

spec의 구성(도입/전개/마무리)을 따라 본문을 작성한다.

**Deep Rewrite 원칙**:

1. **복사 금지** — 블로그 포스트의 문장을 그대로 가져오지 않는다. 같은 아이디어라도 책의 맥락에 맞는 새로운 문장으로 작성한다
2. **톤 전환** — 블로그는 "독자에게 말 거는" 가벼운 톤이지만, 책은 "저자가 사유를 전개하는" 더 깊은 톤. 단, 탐구적 어조("~가 아닐까")는 유지한다
3. **메타포 발전** — 블로그에서 한 문장으로 쓴 비유를 챕터 안에서 발전시키고, 독자가 더 깊이 음미하도록 풀어쓴다
4. **통찰 엮기** — 여러 소스 포스트의 통찰을 엮어 블로그에서는 없던 새로운 논지를 구성한다. 단순 병렬 나열이 아닌 논리적 전개
5. **사례 보존** — 구체적 경험과 사례는 최대한 살린다. 추상적 주장만으로는 독자를 설득할 수 없다

### 5단계: 전환 문장 삽입

spec의 전환 지침을 따라:
- **도입부**: 이전 챕터에서 이어받는 맥락을 자연스럽게 연결
- **마무리**: 다음 챕터로 넘기는 맥락을 암시하거나 질문으로 남김

첫 번째 챕터(프롤로그)는 이전 챕터가 없으므로, 독자의 관심을 잡는 장면이나 질문으로 시작한다.
마지막 챕터(에필로그)는 다음 챕터가 없으므로, 독자에게 여운을 남기는 것으로 마무리한다.

### 6단계: 자체 검토

작성한 초안을 spec과 대조한다:
- 핵심 논지가 명확히 전달되는지
- 소스 포스트의 핵심 부분이 빠짐없이 활용되었는지
- 전환 지침이 반영되었는지
- 예상 분량(3,000~5,000자)에 부합하는지

### 7단계: 산출물 저장

```markdown
# 챕터 {NN}. {제목}

{도입부 — 이전 챕터에서의 전환 또는 독립적 시작}

## {섹션 1 제목}

{본문}

## {섹션 2 제목}

{본문}

## {마무리 제목 또는 무제}

{결론 + 다음 챕터로의 전환}

---
<!-- 소스 포스트: {참조한 포스트 제목 목록} -->
<!-- spec 커버리지: {각 항목 달성 여부 체크} -->
```

**헤딩 규칙**: `#` = 챕터 제목, `##` = 섹션 제목. 블로그의 `###` 규칙과 다르다.

파일명: `_workspace/book/chapters/ch-{NN}-draft.md`

### 8단계: humanize-post 자체검증 (자동)

산출물 저장이 끝나면 humanize-post 스킬을 자동 호출하여 humanize-kr v1.5 quick-rules의 S1 AI-tell 패턴을 자체검증·핀포인트 윤문한다.

**입력**: 7단계에서 저장한 `_workspace/book/chapters/ch-{NN}-draft.md` 경로 (블로그 포스트가 아닌 챕터 파일이지만 본문이 한국어 산문이므로 동일 적용)

**처리**:
- S1 0건 → SKIP (등급 A)
- S1 1~5건 → 자동 핀포인트 윤문 (변경률 10% 이내, 등급 B 진입)
- S1 6건 초과 → 자동수정 **중단**, 사용자에게 글로벌 humanize-kr 정밀 윤문 권고 (`cd ~/.claude/skills/humanize-kr && claude` → "이 글 자연스럽게 윤문해줘")

**Do-NOT 보호**: 인용·고유명사·수치·코드 펜스·spec/소스 메타 코멘트(`<!-- ... -->`)·blueprint의 톤 가이드 준수 voice는 100% 보존. 챕터 deep rewrite의 의도된 비유 발전(메타포 풀어쓰기)은 D-5 의인화 패턴이라도 그대로 둔다.

**재작업 흐름과의 관계**: `revision_feedback` 반영 재작성 후에도 동일하게 8단계 호출. 단, 이미 humanize 통과한 부분이 다시 S1 잔존 0건이면 즉시 SKIP되어 무한 루프는 발생하지 않는다.

**산출물**: `.context/humanize-post/{run_id}/{before.md, findings.json, diff.txt, summary.md}` 자동 생성.

상세 절차·S1 패턴 표·자체검증 6항은 `.claude/skills/humanize-post/skill.md` 참조.

## 재작업 절차

`revision_feedback`이 있는 경우:

1. 기존 `ch-{NN}-draft.md`를 `ch-{NN}-draft-v{N}.md`로 백업 (N은 순번)
2. `revision_feedback`의 지적 사항을 하나씩 확인
3. 수정이 필요한 부분을 수정하되, 수정된 부분에 `<!-- REVISED: {수정 사유} -->` 코멘트 삽입
4. spec 커버리지를 재확인
5. 새 `ch-{NN}-draft.md`로 저장

## 주의사항

- 한 챕터 내에서 같은 사례를 두 번 이상 언급하지 않는다. 다른 챕터에서 같은 사례를 쓰는지는 편집자가 확인한다
- spec에 "추가 집필 필요"로 표시된 부분은 소스 포스트 없이 직접 구성해야 한다. blueprint의 톤 가이드를 따른다
- 프롤로그와 에필로그는 본문 챕터가 모두 완료된 후 작성한다. 본문의 실제 내용을 참조하여 적절한 도입/마무리를 구성하기 위함이다
