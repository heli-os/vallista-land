# 포스트 리뷰 스킬

작성된 블로그 포스트가 프로젝트 규칙을 준수하는지 체크리스트 기반으로 검증한다. 검증 결과를 PASS/WARN/FAIL 세 단계로 구분하여 리포트한다.

## 검증 절차

### 1단계: 대상 파일 확인

포스트 경로를 받거나, 제목으로 검색:
```
packages/blog/content/posts/{제목}/index.md
```
파일이 없으면 content/posts/ 하위를 검색하여 가장 유사한 폴더를 제안한다.

### 2단계: Frontmatter 검증

파일의 YAML frontmatter를 파싱하여 다음을 검증:

| 필드 | 검증 기준 | 실패 등급 |
|------|----------|----------|
| title | 존재 + 비어있지 않음 | FAIL |
| description | 존재 + 비어있지 않음 | WARN |
| image | 값이 `./assets/thumbnail.jpeg` 형식 | WARN |
| tags | 배열 + 각 태그가 허용 목록 내 | FAIL |
| date | `YYYY-MM-DD HH:mm:ss` 형식 | FAIL |
| draft | boolean 값 | FAIL |
| series | 기존 시리즈명과 일치 (필드가 있을 경우) | WARN |

**허용 태그**: 에세이, 기술, 성장, 조직, 스타트업, 회고, 리뷰, 리포트

**기존 시리즈**: 좌뇌의 소설, 생각이 세계가 되는 순간들, 레딧에서는 무슨 이야기를 나눌까?, 볼타 이야기, Vallista-land, JPA, 장인 정신

### 3단계: 본문 구조 검증

frontmatter 이후의 마크다운 본문을 분석:

| 항목 | 검증 기준 | 실패 등급 |
|------|----------|----------|
| 첫 줄 | `![...](./assets/thumbnail.jpeg)` 패턴 매칭 | WARN |
| 섹션 수준 | 본문 내 `#` 또는 `##`가 섹션 제목으로 사용되지 않음 | WARN |
| 최소 섹션 | `###` 헤딩이 2개 이상 | WARN |
| 빈 본문 | frontmatter 이후 실질적 내용 존재 | FAIL |

### 4단계: 파일 시스템 검증

| 항목 | 검증 기준 | 실패 등급 |
|------|----------|----------|
| assets/ | 디렉토리 존재 | WARN |
| 이미지 파일 | assets/ 내 .jpeg, .jpg, .png 파일 존재 | WARN |

### 5단계: 리포트 출력

```markdown
## 리뷰 결과: {포스트 제목}

**경로**: {파일 절대 경로}
**종합**: {PASS / X개 WARN / X개 FAIL}

### FAIL (즉시 수정 필요)
- {항목}: {문제 설명} → {수정 방법}

### WARN (권장 수정)
- {항목}: {문제 설명} → {수정 방법}

### PASS
- {항목}: 통과
```

## 일괄 리뷰

복수 포스트를 리뷰할 때:
1. 대상 파일 목록을 먼저 출력
2. 각 파일에 대해 검증 실행
3. 종합 요약 테이블 출력:

```markdown
| 포스트 | FAIL | WARN | PASS |
|--------|------|------|------|
| {제목} | 0 | 2 | 5 |
```

## 주의사항

- 검증만 수행하고 자동 수정은 하지 않는다. 수정이 필요하면 구체적 수정 방법을 제안
- 레거시 포스트(image가 `1.jpeg`인 경우)는 WARN으로 처리 (기존 포스트의 일괄 변경은 별도 작업)
- description이 없는 레거시 포스트는 WARN으로 처리하되, 새 포스트는 FAIL로 처리
